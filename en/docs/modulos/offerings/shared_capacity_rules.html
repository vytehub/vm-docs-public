Shared Capacity Rules (Offerings)

Coordinación de capacidad entre servicios​

1. Visión general​

Shared Capacity Rules permiten coordinar la disponibilidad de varios Offerings que comparten los mismos recursos humanos.

Estas reglas se configuran durante la creación o edición de un Offering y definen qué otros servicios comparten capacidad cuando coinciden en horario.

Su único efecto es:

Ocultar slots automáticamente cuando la capacidad compartida se agota

No existen acciones adicionales ni cancelaciones automáticas en este tipo de regla.

2. El problema que resuelve​

En escenarios reales:

  • Varias personas pueden ofrecer el mismo servicio
  • Una persona puede ofrecer varios servicios
  • Los servicios tienen duraciones distintas
  • El usuario no quiere asignar personas una por una

Sin reglas de capacidad compartida:

  • Se pueden vender más turnos de los que el equipo puede atender
  • El sistema se vuelve difícil de explicar
  • Aparecen errores operativos

Shared Capacity Rules resuelven esto sin modelar empleados individuales.

3. Principios clave (modelo mental)​

3.1 La capacidad representa personas disponibles​

  • Cada Offering define su capacity
  • Capacity = cuántos turnos simultáneos puede atender ese servicio

3.2 Compartir capacity no cambia la capacity del Offering​

  • Cada Offering mantiene su capacity individual
  • Las reglas solo coordinan cómo se consume esa capacidad en conjunto

3.3 Las reglas son transitivas​

Si:

  • O1 comparte capacity con O2
  • O2 comparte capacity con O3

Entonces:

  • O1, O2 y O3 forman un grupo conectado
  • Todos comparten la misma capacidad efectiva

Aunque el usuario no haya seleccionado O3 directamente desde O1.

4. Configuración en el wizard del Offering (UX)​

Durante la creación o edición de un Offering, el usuario verá el paso:

Capacidad compartida (equipo)​

En este paso puede:

  • Seleccionar una o más Offerings
  • Las Offerings seleccionadas compartirán capacidad cuando coincidan en horario

Ejemplo:

☑ Manicura
☑ Capping
☐ Masajes

5. Normalización automática​

El sistema garantiza que la relación sea bidireccional:

  • Si O1 selecciona O2
  • O2 también queda configurado como compartiendo con O1

El usuario siempre verá el estado correcto desde cualquier Offering.

6. Capacity efectiva compartida​

6.1 Capacity individual​

Cada Offering define su capacity.

Ejemplo:

OfferingCapacity
Manicura2
Capping3
Masajes1

6.2 Capacity del grupo (pool implícito)​

Cuando varios Offerings están conectados por reglas de capacity:

Capacity del grupo = el mayor capacity entre los Offerings conectados

Ejemplo:

  • Capacities: (2, 3, 1)
  • Capacity compartida del grupo = 3

Este valor representa el máximo de personas disponibles para ese conjunto de servicios.

7. Consumo de capacity​

Reglas de consumo​

  1. Cada booking consume 1 unidad
  2. Consume durante:
    • duración del servicio
    • buffers configurados (si existen)
  3. El consumo es global al grupo
  4. Cada Offering también respeta su capacity individual

Fórmula mental​

remainingOffering =
min(
  offering.capacity - bookingsDelOffering,
  pool.capacity - bookingsDelGrupo
)

8. Ejemplo completo — Salón de belleza​

Configuración​

ServicioDuraciónCapacity
Manicura30 min2
Capping45 min3
Masajes60 min1

Offerings conectados por capacity:

  • Manicura ↔ Capping
  • Capping ↔ Masajes

Grupo resultante:

  • Manicura + Capping + Masajes
  • Capacity del grupo: 3

Evolución 9:00–10:00​

Estado inicial

  • Pool disponible: 3
  • Manicura: 2
  • Capping: 3
  • Masajes: 1

Booking 1 — Masajes

  • Pool → 2
  • Masajes → 0 (se oculta)
  • Manicura → 2
  • Capping → 2

Booking 2 — Capping

  • Pool → 1
  • Capping → 1
  • Manicura → 1

Booking 3 — Capping

  • Pool → 0
  • Todos los Offerings → ocultos

9. Filtros avanzados (opcional)​

El usuario puede definir qué bookings descuentan capacity:

  • Status (ej: solo Confirmed)
  • Tags (ej: #internal, #manual)

Usar solo Confirmed puede permitir sobre-reservas en Pending.
Esto es un comportamiento avanzado y responsabilidad del usuario.

10. Qué hacen y qué NO hacen estas reglas​

Hacen​

  • Coordinan capacity entre Offerings
  • Ocultan slots automáticamente
  • Previenen sobre-reservas

NO hacen​

  • No cancelan bookings
  • No notifican
  • No asignan personas
  • No bloquean por eventos externos

11. Resumen final​

Shared Capacity Rules permiten vender solo lo que el equipo realmente puede atender,
sin obligar al usuario a pensar en personas individuales.
