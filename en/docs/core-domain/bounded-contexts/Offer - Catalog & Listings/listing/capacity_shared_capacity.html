Capacity & Shared Capacity

En VyteMerge hay dos formas de “capacidad compartida”:

  1. Pool fijo (Resource Pool): capacidad = número fijo de recursos equivalentes
    (gabinetes, canchas, salas, equipos).
  2. Pool dinámico (Skill/Staff Pool): capacidad depende de quién puede hacer qué y de lo ya reservado
    (ej: estética nails con 3 empleadas y 3 servicios).

Importante: esto es capacidad de recurso (quién/qué puede atender).
No confundir con BookingCapacity (cupos de asistentes) que se usa para Attendees (ej: partido de fútbol 10).

Tipo A: Pool fijo (Resource Pool)​

Cuándo aplica​

  • Gabinetes idénticos: 3 salas de depilación equivalentes
  • Canchas: 2 canchas iguales
  • Equipos: 5 dispositivos iguales
  • “Unidades” indistinguibles

Variables​

  • PoolCapacity = capacidad total del pool (entero >= 1)
  • Interval = intervalo del slot [start, end)
  • BookingsOverlapping(Interval) = bookings que se solapan con el intervalo y consumen el pool
  • Units(b) = unidades consumidas por booking b (default 1)

Solapamiento (overlap)​

Un booking b solapa con Interval si:

b.start < Interval.end y b.end > Interval.start

(Usamos intervalos semiabiertos [start, end) para evitar conflictos en bordes.)

Consumo y decisión​

  • PoolUsed(Interval) = Σ Units(b) para todo b solapado
  • PoolAvailable(Interval) = PoolCapacity - PoolUsed(Interval)

Para una nueva reserva Units(newBooking):

canReserve = PoolAvailable(Interval) >= Units(newBooking)

Si canReserve es false → slot no disponible (conflicto por capacidad).

Tipo B: Pool dinámico (Skill/Staff Pool)​

Cuándo aplica​

Cuando el recurso no es “un número fijo”, sino un conjunto de personas con habilidades:

  • Estética: empleadas con distintas combinaciones de servicios
  • Clínica: médicos con especialidades
  • Staff con certificaciones / skills

En este caso, “capacidad por servicio” es dinámica porque las personas se ocupan.

Ejemplo nails:

  • Empleada 1: A, B
  • Empleada 2: B, C
  • Empleada 3: A, B, C
    Máximos teóricos sin reservas: A=2, B=3, C=2
    Pero la capacidad real cambia según lo reservado en ese horario.

Concepto clave​

No alcanza con “sumar capacities por servicio” porque A, B y C comparten el mismo pool real: las empleadas.

La condición correcta es:

Un nuevo booking es aceptable si existe una asignación posible de los bookings del intervalo a staff que pueda atenderlos.

Eso es un problema de matching (emparejamiento).

Notación​

Para un intervalo I:

  • Staff = conjunto de personas elegibles (empleadas)
  • Skill(e) = servicios que puede hacer la empleada e
  • Bookings(I) = bookings existentes que solapan con I
  • Service(b) = servicio requerido por booking b

Condición de aceptación (matching)​

Existe una función inyectiva:

assign : Bookings(I) ∪ {newBooking} -> StaffAvailable(I)

tal que para todo booking b:

Service(b) ∈ Skill(assign(b))

Si esa asignación existe → ✅ reservable
Si no existe → ❌ no reservable

Equivalente práctico:

maxMatchingSize == numberOfBookingsInInterval

Ejemplo Nails (dinámico)​

Skills:

  • E1: {A, B}
  • E2: {B, C}
  • E3: {A, B, C}

Máximo teórico (sin reservas):

  • Max(A) = 2 (E1, E3)
  • Max(B) = 3 (E1, E2, E3)
  • Max(C) = 2 (E2, E3)

Caso 1: ya hay 2 bookings de A en el mismo intervalo

  • asignación posible: A→E1 y A→E3
  • queda libre solo E2 (B,C)

Entonces en ese mismo intervalo:

  • A disponible: 0
  • B disponible: 1
  • C disponible: 1

Caso 2: ya hay 2 bookings de C en el mismo intervalo

  • asignación posible: C→E2 y C→E3
  • queda libre solo E1 (A,B)

Entonces:

  • A disponible: 1
  • B disponible: 1
  • C disponible: 0

Resultado: aunque “B teórico” sea 3, puede caer a 1 si A o C ya consumieron staff.

Reglas prácticas para el MVP​

¿Qué se valida en runtime?​

  • Pool fijo: validar con la fórmula simple.
  • Pool dinámico: validar con matching sobre el intervalo (staff disponible + skills).

¿Qué bookings se cuentan?​

Recomendación (consistente con evitar overbooking):

  • contar Confirmed + Pending (los que ya bloquearon “tentativamente”)

Revalidación al confirmar​

Siempre revalidar al confirmar booking para evitar race conditions:

  • slots son proyección, booking es verdad.

Cómo convive con “Capacity” de Attendees​

  • BookingCapacity / MinToConfirm / Attendees → cupos de participantes
  • Shared Capacity Pool → capacidad de recursos (staff/salas/equipos)

Ejemplos:

  • Fútbol 5:
    • BookingCapacity=10 (attendees)
    • Pool fijo: cancha (1)
  • Nails:
    • BookingCapacity=1
    • Pool dinámico: staff por skills
