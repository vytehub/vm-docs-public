Price & Promos

TL;DR​

Un Listing es la “pieza comercial” que se muestra y se vende.

  • Un Listing siempre declara precio y moneda (aunque sea “gratis”).
  • Un Listing declara cómo se cobra (pago total, depósito %, depósito fijo).
  • Un Listing puede ser la oferta estándar o una promo (ej: “2x1”).
  • Campaigns promocionan un listing (más visibilidad), pero no cambian el precio por sí mismas.

Regla mental: Catalog define el “qué” (Service/Product), Listing define el “cómo se vende” (precio/políticas).

1) Precio: dónde vive y por qué​

Regla​

El precio de venta vive en el Listing.

  • Service/Product define el “qué” (definición reusable).
  • Listing define el “cómo se vende” (incluye precio, moneda, condiciones y políticas).

Por qué​

El mismo Service o Product puede venderse de distintas maneras:

  • precio normal vs precio promocional
  • precio distinto por canal (VIP vs público)
  • pago completo vs depósito (reducción de no-shows)
  • packs y combos (futuro)

2) Precio base (Base Price) y moneda (Currency)​

El Listing define:

  • BasePrice (número >= 0)
  • Currency (ej: ARS, USD)
  • PricingNotes (opcional): condiciones visibles al usuario (ej: “incluye materiales”, “no incluye traslado”)

Casos típicos​

  • Consulta médica: precio fijo
  • Cancha: precio por hora / franja
  • Servicio a domicilio: precio + adicional por zona (futuro)

Nota: si el precio depende de muchas variables, se modela como “price rules” en el futuro.
En MVP, mantenerlo simple.

3) Payment Policy (cómo se cobra)​

Idea​

El Listing puede exigir pago al reservar (total o parcial) para:

  • reducir no-shows
  • permitir reservas con anticipación
  • alinear expectativas (compromiso)

Payment Policy es una declaración del Listing (producto/negocio).
La integración concreta de pagos vive en Payments/Checkout (módulo/capacidad de plataforma).

Modos recomendados (MVP)​

  • PayNowFull: se paga el total al confirmar
  • DepositPercent: se paga un % como depósito (ej: 30%)
  • DepositFixed: se paga un monto fijo como depósito (ej: $10.000)
  • (futuro) PayLater: se reserva y se paga en persona/externo

Reglas prácticas​

  • El depósito o pago se asocia a la transacción:
    • Service Listing → Booking + (si corresponde) Payment/Order asociado
    • Product Listing → Order
  • El Booking/Order debe guardar snapshot de:
    • BasePrice, Currency, PaymentPolicy y cualquier condición aplicada (para auditoría)

Importante: Payment Policy define cuándo y cuánto se cobra,
no define cómo se procesa (eso lo resuelve la plataforma).

4) Promos: qué son y cómo conviven con un Listing​

¿El listing “es” una promo?​

No necesariamente.

Dos formas válidas:

A) Promo como un Listing separado (recomendación simple)​

Creás un listing adicional para la promo:

  • “Masaje” (listing estándar)
  • “Masaje 2x1 - Promo Agosto” (listing promo)

Ventajas:

  • simple de entender
  • fácil de publicar/archivar
  • se puede promocionar con campaigns sin mezclar reglas

B) Promo como regla dentro del mismo Listing (más avanzado)​

Un listing tiene “promo rules” activas por tiempo/canal. Ventaja: menos listings duplicados
Desventaja: más lógica y edge cases (qué ve quién, cuándo, etc.)

Recomendación MVP: A (promo como listing separado).

5) Options (variantes) vs Add-ons (extras)​

Esta distinción evita confusión.

Options / Variants (cambia “la modalidad” del mismo Listing)​

Una Option es una variante que modifica condiciones del listing sin cambiar el “qué” base.

Ejemplos:

  • “Particular / Obra social / Prepaga” (misma consulta, distinto precio)
  • “Encerado / Pulido” (mismo servicio base, distinta modalidad)
  • “Pack x1 / x3” (mismo producto base, distinta presentación)

En MVP, una option puede:

  • cambiar precio
  • cambiar notas visibles
  • (futuro) cambiar duración/condiciones más complejas

Regla MVP: si una option cambia la duración o el scheduling, conviene otro listing (para evitar edge cases).

Add-ons (se agregan al booking/compra)​

Un Add-on es un extra que se añade a la transacción:

  • Add-on Product: se suma como ítem extra (Order)
  • Add-on Service: se suma como extra de servicio (futuro: extender duración o crear sub-booking)

Ejemplos:

  • “Masaje” + “Crema” (product add-on)
  • “Consulta” + “Electrocardiograma” (service add-on)

Los add-ons se documentan en un archivo aparte (“Add-ons”),
pero se mencionan acá porque impactan precio.

6) Depósitos, no-show y “devolución por compromiso”​

Un listing puede habilitar mecanismos para reducir no-shows:

  • Depósito para asegurar asistencia
  • Devolución / rebate si el usuario asiste (idea de incentivo)
  • Penalidad si no asiste (según política)

Cómo se entiende en el modelo​

  • El Booking bloquea el tiempo
  • El Payment/Order registra el dinero asociado (depósito/fee)

El objetivo no es “castigar”, sino incentivar reservas a largo plazo y mejorar cumplimiento.

7) No confundir con Campaigns​

  • Promos cambian condiciones comerciales (precio/beneficio).
  • Campaigns cambian distribución/visibilidad (más alcance), no necesariamente el precio.

Ejemplo:

  • Listing “Masaje 2x1” (promo)
  • Campaign “Promocionar Masaje 2x1” (publicidad)

Checklist (MVP)​

  • Todo listing declara BasePrice y Currency (o explícitamente “free”)
  • Declarar PaymentPolicy: full / depósito % / depósito fijo
  • Promos simples se modelan como listings separados
  • Options (variantes) ≠ Add-ons (extras) y se documentan como conceptos distintos
  • Booking/Order guarda snapshot de precio/moneda/policy al confirmar/pagar
  • Campaigns se documentan como distribución, no pricing
