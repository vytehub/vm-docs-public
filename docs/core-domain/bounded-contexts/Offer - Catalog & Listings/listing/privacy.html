Privacy

TL;DR​

La privacidad de un Listing tiene dos capas:

  1. Distribución: si aparece o no (Public / Private / Restricted / etc.)
    → se documenta en Channels & Visibility.

  2. Detalle: cuánto se muestra cuando alguien lo ve o lo abre.
    → se controla con Access vía VisibilityMode (None, BusyOnly, BusyOnlyDetails).

El Listing declara intención. Access decide el nivel final de detalle para cada actor.

Objetivo​

Evitar oversharing y permitir que un listing:

  • se descubra sin exponer información sensible
  • degrade su detalle según privacidad/bloqueos/relaciones
  • ofrezca un “preview” controlado (ideal para marketplace/feed)

1) Qué controla el Listing (intención de privacidad)​

El listing puede declarar qué tanto expone, incluso si es Public.

Ejemplos de controles por listing:

  • qué campos se muestran en modo “preview”
  • si muestra “próxima disponibilidad” exacta o solo “hay disponibilidad”
  • si muestra precio exacto o “desde …” (opcional)
  • si permite contacto directo (chat / call-to-action)
  • si muestra ubicación exacta o zona aproximada (futuro)
  • si muestra identidad completa del proveedor o solo marca (según tipo de profile)

Estos controles son “preferencias de presentación”.
No reemplazan el enforcement de Access.

2) VisibilityMode (enforcement por Access)​

Access determina el nivel efectivo de detalle que un actor puede ver.

VisibilityMode.None​

  • No se muestra el listing (o se muestra como “no disponible” según UX).
  • No se muestra disponibilidad.
  • No se permite reservar ni contactar.

Casos típicos:

  • actor bloqueado
  • listing restringido (channel/partner) y actor no cumple
  • privacy/agreements no permiten ver nada

VisibilityMode.BusyOnly​

  • Se permite discovery mínimo:
    • título/preview reducido (según policy)
    • “hay disponibilidad” sin detalles exactos (opcional)
  • No se muestra detalle sensible:
    • horarios específicos
    • info operativa completa
    • datos privados

Casos típicos:

  • el listing se muestra públicamente, pero el owner quiere evitar exposición de horarios exactos
  • el actor no tiene relación suficiente para ver más

VisibilityMode.BusyOnlyDetails​

  • Se muestra el detalle completo permitido para esa experiencia:
    • descripción completa
    • precio completo
    • próxima disponibilidad exacta (si el listing lo permite)
    • botón reservar / comprar
    • contacto (si el listing lo permite)

Casos típicos:

  • owner mirando sus propios listings
  • actor autorizado por follow/agreements
  • listing público con política de full detail

Nota: VisibilityMode no define “qué es un listing”, define “cuánto puede ver este actor ahora”.

3) Cómo se combinan Listing Privacy + Access (regla práctica)​

Pensalo como un mínimo entre:

  • lo que el listing está dispuesto a mostrar (privacy policy)
  • lo que Access permite para ese actor

Ejemplo:

  • Listing quiere mostrar “próxima disponibilidad exacta”
  • Access devuelve BusyOnly
    → se degrada: “hay disponibilidad” (sin horario exacto)

4) Disponibilidad: exacta vs aproximada​

La disponibilidad es uno de los datos más sensibles porque revela patrones.

Opciones recomendadas:

  • Exact: mostrar próximos slots con fechas/horas
  • Approx: mostrar “hay disponibilidad esta semana” o “próximo turno: mañana” sin hora
  • Hidden: no mostrar disponibilidad hasta estar autorizado

Regla:

  • Access define si podés ver None/BusyOnly/Details
  • Listing define si incluso con Details muestra exact o approx

5) Contacto directo (chat) como dato sensible​

Permitir contacto puede aumentar conversión, pero también spam.

Recomendación:

  • AllowContact = false por defecto en listings públicos nuevos (según tu estrategia anti-spam)
  • habilitar contacto para:
    • followers
    • miembros de channel
    • partners
    • usuarios con booking activo

6) Ejemplos típicos​

Listing público en marketplace​

  • En feed: preview mínimo + CTA
  • Al abrir: Access decide detalle según relación/bloqueos

Listing privado por link​

  • Sin link: no aparece
  • Con link: Access igual puede devolver None (blocked) o BusyOnly (preview) o Details

Listing partner-only​

  • Usuario común: None
  • Partner con agreement activo: BusyOnlyDetails

Checklist (MVP)​

  • Definir “preview fields” (qué se muestra en cards)
  • Definir política de disponibilidad: exact/approx/hidden
  • Definir si permite contacto directo
  • Integrar con Access: mapear a VisibilityMode
  • Degradar UI según VisibilityMode (no romper UX)
