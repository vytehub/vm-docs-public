Privacy & Visibility (Users)

Visión general​

La privacidad en VyteMerge define qué puede ver cada persona, en un sistema donde todo se conecta:

  • Profiles
  • Timelines
  • Events
  • Listings
  • Bookings
  • Staff / Partners (Agreements)

La privacidad no es un detalle técnico: es una regla fundacional del producto.

Principio central​

Ver no implica controlar.
Compartir no implica administrar.
Asignar no implica exponer.

La visibilidad siempre se evalúa en contexto.

Qué “define” Users vs qué “decide” Access​

Users define (datos de privacidad del perfil)​

El módulo Users define, por ejemplo:

  • privacidad del Profile (público / privado)
  • qué datos son públicos (foto, bio, etc.)

Social define (relaciones)​

El módulo Social define:

  • follow requests / accepted
  • bloqueos

Access decide (enforcement)​

El módulo Access toma esos inputs y devuelve:

  • Allowed / Denied
  • VisibilityMode: None / BusyOnly / BusyOnlyDetails

Users no decide “quién ve qué”. Users define preferencias.
Access aplica la decisión final.

Privacidad a nivel Profile (modelo tipo Instagram)​

Cada Profile puede estar configurado como:

  • Public
    • cualquiera puede ver el perfil con detalle
  • Private
    • solo followers aceptados ven con detalle
    • no followers ven vista mínima
  • (Opcional futuro) FollowersOnly como estado explícito si lo necesitás

Interpretación con VisibilityMode​

En UX:

  • BusyOnlyDetails → perfil completo (bio, media, links, etc.)
  • BusyOnly → perfil mínimo (nombre + foto + “perfil privado” + CTA seguir)
  • None → no existe / no accesible (bloqueado u oculto)

Relaciones sociales (Follow)​

VyteMerge usa un modelo no simétrico:

  • un usuario puede seguir a otro
  • el otro no tiene que seguir de vuelta

Estados típicos:

  • Requested (si el perfil es privado)
  • Accepted
  • Rejected/Cancelled
  • Unfollowed

Seguir a alguien no otorga permisos operativos.
Solo afecta visibilidad según reglas de Access.

Bloqueos (Block)​

Si A bloquea a B (o viceversa), el resultado esperado es:

  • VisibilityMode.None sobre recursos sensibles
  • y en UX se comporta como “no disponible”

Bloqueo gana sobre follow, sharing o cualquier relación “social”.

Qué pasa con Timelines, Events y Listings​

Users define privacidad del perfil. Los recursos tienen sus propios modelos, pero el patrón se mantiene:

  • Timeline (módulo Timelines): privacy del timeline + sharing
  • Listing (módulo Listings): intención de visibilidad (public/private/channels)
  • Event/Booking: son instancias en el tiempo

En todos los casos:

Access aplica el “cuánto” (VisibilityMode) y los permisos de acción.

Importante: Sharing no es Delegation​

Timeline Sharing​

Es visibilidad del tiempo. Sirve para:

  • coordinación familiar/equipos
  • evitar conflictos
  • mostrar ocupado sin exponer detalles

Se expresa con:

  • VisibilityMode.BusyOnly o BusyOnlyDetails

No debería implicar:

  • crear/editar events
  • reagendar bookings
  • publicar listings

Delegation (Staff) por Agreements​

Si alguien necesita operar (gestionar agenda, bookings, publicación):

  • eso es un Delegation Agreement
  • con permisos explícitos (ManageSchedule, ManageBookings, Publish, etc.)
  • y Access aplica esas capacidades

Ejemplo (perfil privado tipo Instagram)​

  • Perfil público → BusyOnlyDetails
  • Perfil privado + follower aceptado → BusyOnlyDetails
  • Perfil privado + no follower → BusyOnly
  • Bloqueado → None

Frase final​

La privacidad en VyteMerge no es esconder información,
es compartir solo lo necesario, en el contexto correcto.
