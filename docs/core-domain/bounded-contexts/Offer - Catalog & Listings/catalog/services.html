Services

TL;DR​

Un Service es la definición reusable de qué ofrecés.

  • Vive en el Catalog.
  • Es estable y reutilizable.
  • No define precio, promos, canales ni scheduling final.
  • Se vende/reserva a través de Listings.

Regla mental:
Service = qué es
Listing = cómo se ofrece / vende / agenda

Qué es un Service​

Un Service describe un servicio de manera reusable y relativamente estable.

Ejemplos:

  • Consulta médica
  • Masaje descontracturante
  • Alquiler cancha fútbol 5
  • Depilación láser
  • Sesión de terapia

Un Service existe para:

  • mantener una definición única (“Masaje”)
  • permitir múltiples Listings (promo, express, canal, etc.)
  • reportar demanda por tipo de servicio (analytics)
  • facilitar reutilización (fotos base, descripción base, tags)

Qué problemas resuelve (por qué no usar solo Listings)​

Sin Services, terminás duplicando el mismo “qué” en múltiples ofertas:

  • “Masaje 60m”
  • “Masaje VIP”
  • “Masaje Promo 2x1”
  • “Masaje Express”

Eso genera:

  • descripciones inconsistentes
  • media duplicada
  • tags duplicados
  • reportes difíciles (“¿cuántos masajes vendo?”)

Con Services:

  • la definición vive una sola vez
  • cada oferta se expresa como un Listing

Propiedades típicas (lenguaje de negocio)​

Campos comunes:

  • Name (nombre)
  • Description (descripción)
  • Category (opcional)
  • Tags (para discovery)
  • BaseDuration (duración base)
  • BaseMedia (fotos, cover)
  • Requirements / Notes (opcional)
    Ej: “venir en ayunas”, “traer estudios”, “no usar crema el día anterior”
  • BaseTerms (opcional)
    Términos genéricos del servicio (no de la oferta puntual)

Duración base (qué significa)​

La duración base representa el tiempo típico del servicio en condiciones normales.

No es necesariamente la duración final porque:

  • un Listing puede vender una versión express (30m)
  • un Listing puede agregar buffers antes/después
  • un Listing puede ser grupal (y variar duración o dinámica)
  • un Listing puede cambiar duración por canal (ej: VIP más largo)

Regla recomendada

  • Service guarda duración base (semántica)
  • Listing define duración efectiva + buffers + reglas operativas/comerciales

Media (fotos) y relación con Listings​

  • El Service puede tener media base (default).
  • El Listing puede tener media propia (promo, flyer, campaña).

Regla de UI (MVP)

  1. Si el Listing tiene media → se muestra la del Listing.
  2. Si no → fallback a media del Service.

Esto permite:

  • mantener fotos “oficiales” del servicio
  • sin impedir promos específicas por listing

Relación con Place (dónde ocurre)​

Un Service puede ser:

  • agnóstico (sin lugar)
  • o tener un “lugar típico” como referencia

Pero la verdad del “dónde” se define en:

  • el Listing (ej: “Consulta en consultorio”, “Visita a domicilio”)
  • o el Booking/Event (instancia concreta, dirección final)

Recomendación MVP: mantener Place a nivel Listing/Booking, no en Service.

Ejemplos de modelado​

1) Médico con hospital + consultorio​

  • Service: Consulta
  • Listing A: “Consulta consultorio”
    • Place = consultorio
    • duración = 30m
  • Listing B: “Consulta hospital”
    • Place = hospital
    • duración = 20m

El Service no se duplica. Cambia la oferta.

2) Estética nails (3 servicios)​

Services:

  • Capping (90m base)
  • Manicura (60m base)
  • Gel (30m base)

Los Listings definen:

  • pricing/promos
  • scheduling (buffers, ventana, etc.)
  • capacity pools y assignment (staff)

Qué NO debe ir en Service (errores comunes)​

  • ❌ Precio y promos (van en Listing)
  • ❌ Visibilidad/canales (van en Listing/Channels)
  • ❌ Reglas de agenda/conflictos del usuario (van en Timeline Conflict Rules)
  • ❌ Auto-asignación/capacity pools (van en Listing/Assignment)
  • ❌ Formularios de booking (van en Listing → Forms)

Checklist (MVP)​

  • Service tiene Name/Description/Tags/BaseDuration
  • Service puede tener media base (fallback)
  • Listing referencia Service y define el “cómo”
  • Duración base ≠ duración efectiva (listing manda)
