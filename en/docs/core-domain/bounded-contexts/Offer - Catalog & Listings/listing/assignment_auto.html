Assignment (auto-asignación)

TL;DR​

Assignment permite que un negocio venda un Service Listing y que el sistema asigne automáticamente cada booking a una persona (staff) o recurso, sin que ese staff tenga que “manejar el catálogo” ni publicar listings.

  • El negocio controla qué se vende (Listing).
  • El sistema decide quién atiende (Assignment).
  • El resultado se refleja en el Timeline del responsable (Event asignado).

Staff y permisos se gobiernan por Agreements + Access. Assignment solo decide “a quién le cae”.

Contexto / Problema​

En negocios con staff (estética, clínica, gimnasio, consultorio compartido):

  • el negocio quiere un único punto de venta (un listing por servicio)
  • pero la atención la realizan personas distintas
  • y esas personas necesitan ver su agenda para organizarse

Sin Assignment, el negocio tendría que:

  • crear listings duplicados por cada empleado, o
  • asignar manualmente cada booking, o
  • darle control comercial al staff (lo cual no siempre quiere)

Objetivo de negocio​

  • Evitar fricción operativa (menos “administración manual”).
  • Mantener una sola “oferta” comercial (listing) y asignar internamente.
  • Garantizar que el booking respeta:
    • disponibilidad real (conflicts)
    • shared capacity (pools)
    • skills/limitaciones del staff
  • Soportar escenarios:
    • nails con skills distintos
    • clínica con especialidades
    • gabinetes + profesionales
    • partner/reseller que vende pero no atiende

Qué es Assignment exactamente​

Assignment es una decisión que, dado un Booking (o intento de booking), resuelve:

  • AssigneeType: StaffMember | Resource | Pool
  • AssigneeId: quién es el responsable final
  • AssignmentReason: por qué se eligió (audit/debug)

Luego:

  • se crea/actualiza un Event en el Timeline correspondiente del assignee (o se asigna el Event del owner al assignee según tu modelo)
  • se registra el vínculo en el booking (AssignedTo)

Modelo mental (simple)​

Un booking reserva un intervalo de tiempo. Assignment decide quién lo consume.

  • En nails: el booking consume 1 empleada durante el intervalo.
  • En gabinete: el booking consume 1 gabinete (pool fijo), y opcionalmente 1 profesional (pool skills).

Ejemplo Nails (skills pool)​

Empleadas:

  • E1: puede A, B
  • E2: puede B, C
  • E3: puede A, B, C

Flujo:

  1. el negocio publica un listing “Servicio B”
  2. entra un booking para B en [10:00, 11:00)
  3. Assignment elige una empleada disponible que pueda B
  4. el event aparece en el Timeline de esa empleada

Resultado:

  • el negocio vendió “B”
  • el sistema asignó “quién atiende”
  • se evita duplicar listings por empleada

Inputs que usa Assignment​

1) Del Listing​

  • AssignmentEnabled (bool)
  • AssignmentMode (por ejemplo Auto | Manual)
  • EligibilityRules (quién puede recibir asignación)
    • skills requeridos (service)
    • tags/categorías (opcional)
  • Strategy (cómo elegir dentro del grupo elegible)
  • Constraints (opcional)
    • “solo mujeres”, “solo senior”, “solo X lugar” (si aplica)
  • SharedCapacityPoolId (si el listing consume un pool fijo)

2) Del Booking​

  • Interval (start/end)
  • ServiceId (o service extension si add-ons)
  • PlaceId (si aplica)
  • AddOns (si extienden duración)

3) Del contexto operativo​

  • disponibilidad real en Timeline (conflicts)
  • pools de capacidad (fijo o skills)
  • bloqueos/permisos (Access) si influyen en la asignación
    • ej: staff revocado por agreement no es elegible

Estrategias de asignación (lenguaje de negocio)​

Round-robin​

Distribuye equitativamente para balancear carga.

  • útil para estética (varias empleadas similares)

Skills-first​

Primero filtra por skills, luego elige por disponibilidad.

  • útil cuando no todos pueden hacer todo

Priority / Ranking​

Lista con prioridad (E3 primero, luego E1, etc.).

  • útil para “senior” o responsables principales

Least-loaded​

Elige el que tenga menos bookings en un período.

  • útil para distribuir trabajo sin rotación estricta

Recomendación MVP: empezar con Skills-first + Round-robin dentro del grupo.

Reglas de disponibilidad (qué puede fallar)​

Un assignee es elegible solo si:

  • está disponible en el intervalo (no hay conflicto)
  • pertenece al grupo habilitado (agreement activo)
  • cumple skills requeridos
  • no rompe shared capacity (si aplica)

Si no hay candidato:

  • el booking no puede confirmarse (o queda Pending/Requested según tu política)
  • se devuelve razón clara: NoEligibleAssignee

Assignment y cambios posteriores​

Re-asignación (futuro / opcional)​

Puede ocurrir si:

  • el staff cancela
  • el negocio decide re-balancear
  • se detecta conflicto nuevo

Política recomendada:

  • permitir re-asignación solo antes de un deadline
  • registrar auditoría (quién, cuándo, por qué)

Lock-in al confirmar​

Una vez Confirmed, el booking debería:

  • persistir AssignedTo
  • no cambiar silenciosamente (a menos que haya re-asignación explícita)

Interacción con Agreements + Access​

Assignment no gestiona permisos. Solo debe respetar “quién es elegible” según relaciones vigentes.

Ejemplos:

  • si una empleada sale del staff (agreement revocado) → deja de ser elegible
  • si un partner vende un listing pero no atiende → no entra en elegibles de assignment

KPIs (para producto/operación)​

  • % de bookings auto-asignados exitosamente
  • % de fallos por “no hay staff elegible”
  • distribución de carga por staff (balance)
  • tiempo promedio hasta confirmación (si hay pending/request)

Checklist (MVP)​

  • AssignmentEnabled por listing
  • definir pool de elegibles (staff con agreement activo)
  • filtrar por skills (service) y disponibilidad (conflicts)
  • estrategia simple (round-robin dentro de elegibles)
  • persistir AssignedTo en booking al confirmar
  • crear/asignar Event en timeline del staff
  • razones claras de error (NoEligibleAssignee, Conflict, PoolFull)
