Timeline Conflict Rules

Decisiones automáticas ante conflictos de agenda​

1. Visión general​

Las Timeline Conflict Rules permiten definir cómo el sistema debe reaccionar cuando existen conflictos reales de agenda.

Un conflicto ocurre cuando:

  • existe un Event (compromiso real)
  • se superpone en el tiempo
  • con slots proyectados o con reservas por ocurrir

Estas decisiones:

  • viven en el Timeline
  • son independientes de un listing específico
  • se aplican a múltiples ofertas y contextos
  • centralizan cómo el sistema reacciona ante la realidad

A diferencia de Capacity Pools (que controlan cuántas cosas pueden ocurrir), las Timeline Conflict Rules controlan si algo debería mostrarse o permitir reserva.

2. El problema que resuelven​

En la vida real:

  • lo personal, familiar y laboral se mezcla
  • no todos los eventos afectan todo lo que vendo
  • si “desaparecen slots” sin explicación, se pierde confianza

Las Timeline Conflict Rules permiten expresar estas decisiones una sola vez, en un solo lugar.

3. Principios clave​

3.1 Se observan events, no intenciones​

  • no asignan staff
  • no “entienden” por qué un evento existe
  • solo observan Events con filtros

3.2 El Timeline decide​

  • el Listing define reglas operativas/comerciales (duración, buffers, repetición, etc.)
  • el Timeline define qué cuenta como conflicto y qué hacer

3.3 Separación clara​

  • Slots → disponibilidad proyectada (no verdad)
  • Events → compromisos reales
  • Conflicts → decisiones ante solapamientos

4. Anatomía de una rule (Source / Target / Action)​

Cada rule se define con tres partes:

  1. Source – qué events observar
  2. Target – qué se ve afectado
  3. Action – qué decisión tomar

5. Source — Events observados​

Timelines observados​

Una rule puede observar events en:

  • el Timeline actual
  • otros timelines del mismo Profile
  • timelines compartidos (familia, colegio, etc.)

Filtros recomendados​

Status​

  • Pending
  • Confirmed

Cancelled y Completed se ignoran por defecto.

Tags​

Ejemplos:

  • #travel
  • #family
  • #school
  • #impostergable

6. Importante: si el source es un timeline compartido​

Si querés que una conflict rule observe tags en un timeline compartido:

  • el sharing debe ser al menos BusyOnlyDetails (porque BusyOnly no expone tags)
  • o la rule debe basarse solo en “ocupación” (BusyOnly)

Esto evita falsas expectativas del tipo “quiero filtrar por #school pero no compartimos tags”.

7. Target — Qué se ve afectado​

Listings reservables​

  • listings específicos
  • o All Listings reservables
  • o listings que cumplan una condición simple (ej: “presencial” vs “remoto”, categoría, tag del listing)

Nota: esto reemplaza “Offerings” del modelo anterior.

Events (opcional / futuro)​

  • notificar sobre events
  • o aplicar acciones (con mucha cautela)

8. Actions — Decisiones del sistema​

Sobre Slots (MVP)​

Hide slots (default)​

  • el slot no se muestra
  • no afecta bookings existentes

Notify only​

  • el slot puede mantenerse visible (según política)
  • se genera alerta/revisión

Sobre Events (futuro)​

Cancel event / Cancel and reschedule​

  • solo cuando exista un marco claro de permisos, auditoría y UX
  • recomendable dejarlo como fase futura

9. Cuándo se activa un conflicto​

Una rule se ejecuta cuando:

  • existe un event observado
  • cumple filtros
  • se solapa temporalmente
  • considerando duración y buffers (si aplica)

10. Relación con Capacity Pools​

Orden mental recomendado:

  1. Timeline Conflict Rules (¿es elegible el intervalo?)
  2. Capacity Pools (¿hay cupo real en ese intervalo?)
  3. Mostrar u ocultar slots

Si hay conflicto → no tiene sentido evaluar capacity.

11. Ejemplos reales​

Bloqueo total por viaje​

Source

  • Timeline personal
  • Tag #travel

Target

  • All Listings reservables

Action

  • Hide slots

Evento escolar afecta solo listados presenciales​

Source

  • Timeline familiar
  • Tag #school

Target

  • Listings “presenciales”

Action

  • Hide slots

Monitoreo sin bloqueo automático​

Source

  • Timeline de equipo/organización
  • Status Confirmed

Target

  • All Listings reservables

Action

  • Notify only

12. UX sugerido​

  • Tab “Conflicts” dentro del Timeline
  • reglas expresadas en lenguaje natural:

“When timeline X has event tagged #school → hide slots for Listings presenciales”

  • vista previa del impacto

13. Frase final​

Timeline Conflict Rules no venden tiempo.
Aseguran coherencia cuando la realidad se superpone.
