Arquitectura Técnica

1) Contexto & Objetivo​

Definir la arquitectura para el MVP Web de VyteMerge, maximizando time-to-market con .NET 8 + Angular y aislando desde el inicio el servicio de IA + Search (IR) para experimentar rápido en Search/Feed sin frenar el desarrollo del dominio.

  • Equipo: 1 Arquitecto de software, 1 dev Jr, 1 diseñadora Jr.
  • Objetivo técnico: Modular Monolith + CQRS (MediatR) (.NET) para dominio y Servicio IR separado para búsqueda y recomendaciones.
  • Objetivo de producto: publicar Offering/Slot/Booking, Social básico (Profiles/Follows/Posts), Campaigns, Search + Feed.

2) Alcance (In/Out)​

Incluye (MVP)

  • Web Angular (repos separados), design system (vt-toolkit, vt-shared) con Storybook.
  • Backend Modular Monolith .NET 8 + CQRS (MediatR), BFF Minimal APIs, gRPC interno (donde aplique).
  • IR Service (IA + Search) expuesto vía /search y /feed.
  • PostgreSQL (core), Redis (cache), Blob Storage (archivos), Service Bus (eventos).
  • Autenticación JWT corto + refresh; niveles de privacidad (cuenta/timeline/slot).
  • Observabilidad: OpenTelemetry + App Insights.

Fuera de alcance (Fase 2)

  • SSR/PWA, app mobile nativa, sync bidireccional con calendarios externos, escrow/hold en pagos, streaming avanzado (Event Hubs/Kafka).

3) Requisitos​

Funcionales

  • Offerings, Slots, Bookings; Profiles, Follows, Posts; Campaigns.
  • Search con filtros (tiempo/geo/categoría/visibilidad) y Feed personalizado.

No funcionales (SLOs)

  • API lecturas p95 ≤ 300ms; escrituras críticas p95 ≤ 600ms.
  • IR/Search p95 ≤ 300ms; index freshness < 10s tras publicar/actualizar Slot.
  • Disponibilidad objetivo MVP: 99.5%.
  • Seguridad: TLS 1.2+, secretos en Key Vault; logging/trazas E2E.

Frontend (repos separados)

7) IR Service (IA + Search)​

Decisión pendiente: lenguaje/stack, motor de búsqueda/vector, embeddings y ranking.

Propuesta por defecto (costo‑beneficio‑velocidad):

  • Lenguaje: Python + FastAPI (iteración ML rápida; skills existentes).
  • Búsqueda: Azure AI Search (BM25 + Semantic Ranker + Vector en el mismo servicio).
  • Embeddings: Azure OpenAI (batch; no al vuelo) con versionado.
  • Ranking: híbrido (reglas + pesos) en MVP; luego LightGBM/XGBoost.
  • Señales: Monolito → Service Bus → IR (views, clicks, bookings, follows).
  • SLOs: p95 /search ≤ 300ms; freshness <10s.

Contratos frontera (BFF → IR)

GET /search?q&when&near&category&cursor&limit&fields
GET /feed?userId&near&timebox&cursor&limit&fields
POST /feedback { itemId, type: "impression|click|dismiss|save|book" }

Provider Abstraction (para no perder control)

  • SearchProvider dentro de IR para permitir reemplazar AI Search por Elastic + pgvector sin tocar FE/BFF.
  • EmbeddingProvider (Azure OpenAI u OSS).

Indexación & Señales

  • Indexador IR suscripto a eventos: slot_published|updated|booked|cancelled, follow_created, post_published, card_clicked.
  • Embeddings en batch por cambios sustanciales; Redis para features online (CTR reciente, afinidad social, popularidad).

9) API (extracto MVP)​

Auth/Profiles

  • POST /auth/login • POST /auth/signup
  • GET /profiles/:id • PATCH /profiles/:id

Follow

  • POST /follows • PATCH /follows/:userId/(approve|remove|block)
  • GET /profiles/:id/(followers|following)

Offerings/Slots

  • POST /offerings • PATCH /offerings/:id
  • POST /slots/generate (reglas → slots)
  • GET /slots?timelineId=&visibility=public

Bookings

  • POST /bookings
  • PATCH /bookings/:id/(confirm|cancel)

Posts/Campaigns

  • POST /posts • DELETE /posts/:id
  • POST /campaigns • GET /campaigns?owner=...

Discovery / IR

  • GET /search?q&when&near&category&cursor&limit&fields
  • GET /feed?userId&near&timebox&cursor&limit&fields

Versionado: header Accept: application/vnd.vytemerge.v1+json.

10) Seguridad & Privacidad​

  • Auth: JWT corto + refresh; scopes/roles.
  • Privacidad: cuenta/timeline/slot; filtros de visibilidad en IR/Search.
  • Cifrado: TLS 1.2+; secretos en Key Vault.
  • PII: mínimo necesario; logs sin datos sensibles; retention acorde.

11) Observabilidad​

  • Trazas: OpenTelemetry (traceId de FE→BFF→Monolito/IR→AI Search).
  • Métricas: p95 /search, /feed; tasa de error; colas atrasadas.
  • Logs: Serilog/estructura; App Insights con sampling (10–20% en MVP).

12) Despliegue & Entornos​

  • Azure + AKS (MVP con 1–2 réplicas por servicio).
  • Entornos: dev • staging • prod.
  • Front Door (CDN/WAF), dominios y TLS administrados.
  • CI/CD: GitHub Actions (build/test/lint → deploy por entorno; trunk-based + feature flags).

13) SLOs & KPIs​

  • SLO: /search y /feed p95 ≤ 300ms; freshness <10s.
  • KPIs: CTR feed, conversión Slot→Booking, utilización timeline, aceptación follow, ROAS campañas.

14) Plan de Evolución​

  1. MVP: ranking por reglas; embeddings batch; cache Redis; Service Bus básico.
  2. Fase 1.5: ranker LightGBM/XGBoost; A/B simple; más señales (dwell/saves).
  3. Fase 2: Event Hubs/Kafka; feature store ligero; SSR en páginas públicas (opcional); mobile con Angular + Capacitor.

15) ADRs (resumen)​

  • ADR-001: Término Service → Offering (global).
  • ADR-002: Monolito modular + BFF con IR Service separado desde día 1.
  • ADR-003: PostgreSQL como DB Core.
  • ADR-004 (Propuesta): IR Python + AI Search (texto+vector) + Embeddings Azure OpenAI; ranking híbrido; SLOs definidos.

16) Riesgos & Open Questions​

  • IR Stack (Python vs .NET) y motor de búsqueda (AI Search vs Elastic + pgvector) → Propuesta arriba.
  • Costo AI Search/embeddings a escala → mitigado con embeddings batch + cache + escalado gradual de SKU.
  • gRPC interno con equipo chico → mantenerlo en BFF↔Monolito; evitar gRPC entre todos los módulos al inicio.
  • Repos separados FE → disciplina para compartir modelos/SDKs; opción de generar SDK desde OpenAPI.

17) Test / Acceptance Criteria​

  • GET /search y GET /feed cumplen p95 ≤ 300ms en staging con dataset MVP.
  • Indexación < 10s tras slot_published.
  • Outbox + Service Bus con reintentos e idempotencia.
  • Observabilidad con traceId E2E y dashboards básicos.
  • Build/Deploy verde a dev y staging desde CI.

18) Checklist / Next Steps​

  • Confirmar IR stack (aceptar Propuesta o ajustar).
  • Definir esquema de índices y feature set inicial en IR.
  • Escribir OpenAPI de /search, /feed, /feedback.
  • Implementar Indexador (Service Bus → IR → AI Search).
  • Preparar pipelines CI/CD (BFF, Monolito, IR) y secretos en Key Vault.
  • Crear épica Platform Setup con IaC y provisión de: AKS, AI Search, Redis, Key Vault, App Insights, Front Door.
