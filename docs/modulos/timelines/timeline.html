Timeline

Visión general​

El Timeline es uno de los conceptos centrales de VTWA.
No es un calendario tradicional, sino un motor semántico de tiempo que describe:

  • Qué eventos existen
  • Qué slots pueden ofrecerse
  • Qué reglas afectan la disponibilidad
  • Cómo se coordinan múltiples agendas

El Timeline es el punto donde convergen:

  • Offerings
  • Bookings
  • Events
  • Availability Rules (documentadas aparte)

Qué es un Timeline​

Un Timeline representa la agenda de una entidad.

Puede representar:

  • Una persona
  • Un profesional
  • Un recurso
  • Un contexto (familia, equipo, organización)

Cada Timeline:

  • Tiene un owner
  • Tiene eventos propios
  • Puede ser compartido (Phase 2)
  • Puede observar otros Timelines mediante reglas

Múltiples Timelines por usuario​

Un usuario puede crear múltiples Timelines.

Esto permite modelar escenarios reales sin forzar estructuras artificiales.

Ejemplos​

  • Consultorio médico:
    • Timeline por cada médico
  • Centro de estética:
    • Timeline por profesional
  • Usuario individual:
    • Timeline personal
    • Timeline familiar
    • Timeline laboral

Cada Timeline es independiente, pero puede coordinarse con otros.

Qué contiene un Timeline​

Un Timeline puede contener:

  • Events
    • Bookings
    • Bloqueos manuales
    • Eventos externos
  • Slots
    • Derivados de Offerings
    • Sujetos a reglas
  • Policies
    • Resultado de compilar reglas
    • No visibles directamente para el usuario

El Timeline NO es un calendario tradicional​

Aunque maneja tiempo, el Timeline no se representa como una grilla fija.

Diferencias clave​

CalendarioTimeline
Grilla por díasFlujo continuo
Visual primeroSemántico primero
Un contextoMúltiples contextos
Horas fijasRelaciones temporales

El Timeline cuenta una historia temporal.

Modelo mental recomendado​

Pensar el Timeline como:

Una sucesión de eventos y oportunidades en el tiempo, explicadas por reglas.

No como una tabla de horas.

Visualización conceptual (futuro)​

El Timeline puede representarse visualmente de distintas formas.

Timelines en paralelo​

  • Varios Timelines uno debajo del otro
  • Permite ver conflictos y coordinación

Conexiones temporales​

  • Líneas que conectan eventos coincidentes
  • Explican bloqueos y dependencias

Inspiración tipo Git merge​

  • Cada Timeline como una branch
  • Los eventos como commits
  • Las reglas generan merges, bloqueos y exclusiones

Esta metáfora ayuda a entender el sistema, aunque no define la UI final.

Relación con Availability​

El Timeline no define reglas, solo las ejecuta.

  • Las reglas viven en módulos específicos
  • El Timeline compila policies
  • Las policies determinan qué slots y eventos se muestran

(Ver documentación de Availability por separado)

Frase final​

El Timeline no es un calendario.
Es el motor que explica qué puede pasar en el tiempo y por qué.
