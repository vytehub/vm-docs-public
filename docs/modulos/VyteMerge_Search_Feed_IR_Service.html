VyteMerge — Search & Feed (IR Service) — Diseño & Decisión

1) Contexto / Objetivo​

Objetivo: definir el diseño específico de Search & Feed para VyteMerge con foco en costo–beneficio–velocidad sin perder control sobre la relevancia.
Decisión principal: Servicio IR (Intelligence & Retrieval) en Python + FastAPI que orquesta candidatos y re-ranking, utilizando Azure AI Search como motor gestionado de indexado, BM25 y vector search. Embeddings generados offline (batch) con Azure OpenAI. Contratos REST estables (/search, /feed, /feedback) expuestos por IR y consumidos por el BFF .NET. Señales vía Service Bus.

Razonamiento: “buy” lo infra‑pesado (índice, ANN, sharding) y “build” lo diferencial (features, ranking, mezcla de señales). Mantiene salida rápida y control futuro (proveedor pluggable).

2) Alcance​

Incluye

  • IR Service (Python FastAPI): retrieval híbrido (texto + vector), re‑ranking, features online (Redis), APIs públicas de Search/Feed, indexador por eventos.
  • Azure AI Search: índices offering, slot, profile, post con campos texto, facets y vector.
  • Azure OpenAI: embeddings batch en ingesta.
  • Redis: caché de consultas/facets y features (contadores recientes, afinidad).
  • Azure Service Bus: eventos de dominio y señales de comportamiento.
  • BFF .NET: fachada estable para Web (y futuro Mobile).

No incluye

  • Operación de Elastic/pgvector/Qdrant on‑prem (se deja plan de migración).
  • Modelos neurales pesados en línea (se inicia con reglas + LGBM/XGBoost).

3) Requisitos​

Funcionales​

  • /search: consulta híbrida (BM25 ⊕ semántico) con filtros (geo, tiempo, categoría, privacidad).
  • /feed: recomendaciones personalizadas por usuario y contexto (near/timebox).
  • /feedback: recepción de señales (opcional si no viajan por eventos).
  • Actualización de índices < 10s tras publicar/editar un Slot.

No funcionales (MVP)​

  • p95 /search y /feed ≤ 300 ms (staging, sin imágenes).
  • Disponibilidad ≥ 99.5%.
  • Observabilidad E2E (traceId FE→BFF→IR→AI Search/Redis).
  • Privacidad: respeto de visibilidad por cuenta/timeline/slot y opt‑in para datos de localización.

4) Diseño​

4.1 Arquitectura (alto nivel)​

Notas clave

  • IR define esquema lógico, mezcla BM25+vector y aplica re‑ranking con features propias (geo/tiempo/social/popularidad/campañas).
  • AI Search maneja indexado, particionado, réplicas y ANN. IR es dueño del ranking final.
  • Provider pluggable: IR implementa una interfaz SearchProvider → AzureSearchProvider (hoy) / ElasticProvider (futuro).

7) Ranking (MVP → Fase 1.5)​

Score final (MVP — reglas)

score = wb*bm25 + ws*vectorSim + wg*geoScore + wt*timeFit + wa*affinity + wp*popularity + wc*campaignBoost
  • bm25: score de texto (AI Search).
  • vectorSim: coseno entre embedding consulta y doc.
  • geoScore: decae con distancia.
  • timeFit: cercanía al hueco de usuario (si hay when).
  • affinity: afinidad social (sigo al owner, seguidores en común).
  • popularity: CTR reciente (Redis).
  • campaignBoost: multiplicador de campañas.

Fase 1.5 — LGBM/XGBoost

  • Features: las anteriores + price_norm, duration_norm, topic_match, freshness, device_type, hour_bucket.
  • Entrenamiento offline (Python). Serving en IR (Python) o export ONNX si migrase a .NET.
  • A/B simple via feature flag en BFF.

8) Ingesta & Señales​

8.1 Eventos de dominio (Service Bus)​

  • slot_published|updated|archived
  • slot_booked|booking_cancelled
  • follow_created|removed
  • post_published
  • campaign_impression

Ejemplo payload

  "event": "slot_published",

  "ownerId": "u_biz1",

  "visibility": "public",
  "ts": "2025-08-20T10:00:00Z"
}

8.2 Señales de comportamiento​

  • slot_viewed, card_clicked, item_saved, booked.
  • Ingesta por /feedback o eventos. Agregación en Redis + lotes a almacenamiento frío si se requiere analítica.

9) Seguridad & Privacidad​

  • Respeto de visibilidad: filtros por visibility, timelineVisibility, y scope de usuario en consultas a AI Search.
  • Opt‑in de localización: sólo aplicar near/geoScore si user consiente.
  • Cifrado: secretos en Key Vault, TLS end‑to‑end.
  • PII mínima en índices (referenciar ids, no datos sensibles).

10) Costos (resumen práctico)​

  • Driver principal: Azure AI Search (SKU/replicas/partitions) + tokens de embeddings (batch).
  • Palancas: S1 al inicio (1 réplica), cache Redis, embeddings offline, facets precomputadas populares.
  • Compute IR: 1–2 pods chicos. DB core pequeño (SQL/PG). Service Bus Basic/Standard. App Insights con sampling 10–20%.

11) KPIs & Observabilidad​

  • p95 /search, /feed; error rate; freshness (<10s).
  • CTR de feed/search, NDCG@10 (si aplicable), % resultados vacíos.
  • Trazas OTel con traceId propagado (FE→BFF→IR→AI Search/Redis).
  • Dashboards: latencia por filtro, hit ratio Redis, costo AI Search.

12) Roadmap de Evolución​

  1. MVP: reglas + mezcla BM25/Vector; embeddings batch; Redis cache; eventos básicos.
  2. Fase 1.5: ranker LGBM/XGBoost, más señales, A/B simple, pesos por categoría.
  3. Fase 2: Event Hubs/Kafka (si crece analítica), feature store ligero; posibilidad de migrar provider (Elastic+pgvector) sin tocar FE/BFF.

13) ADR — IR‑001 (Decisión de Arquitectura)​

  • Decisión: IR Service en Python FastAPI con Azure AI Search; embeddings batch con Azure OpenAI; ranking propio.
  • Alternativas: .NET IR con AI Search; Elastic+pgvector self‑hosted.
  • Motivo: mejor velocidad de experimento y control de ranking manteniendo bajo costo operativo.
  • Estado: Aprobado (MVP). Revisión en Fase 1.5.

14) Test / Acceptance Criteria​

  • /search y /feed p95 ≤ 300 ms (staging), error rate < 1%.
  • Freshness indices < 10s tras slot_published.
  • Cache hit en Redis ≥ 40% para queries populares.
  • Privacidad: consultas respetan visibilidad y opt‑in geo (tests).
  • Observabilidad: trazas completas con traceId, métricas y logs accionables.

15) Riesgos & Open Questions​

  • Costos si crece AI Search → monitorear uso/replicas; cache/facets.
  • Sesgos en ranking → logging de contribución de features; validación offline.
  • Vendor lock‑in → SearchProvider pluggable; pruebas con Elastic/pgvector.
  • GPU/Modelos pesados (futuro) → separar servicio de inferencia y orquestar desde IR.

✅ Checklist / Next Steps​

  • Definir versión de embeddings (embedding_v1) y dimensiones.
  • Crear índices en Azure AI Search (offering/slot/profile/post).
  • Implementar IR Service (FastAPI) con endpoints /search, /feed, /feedback.
  • Implementar Indexador (Service Bus → IR → AI Search).
  • Añadir Redis (cache + features online).
  • Configurar OTel + App Insights y alertas de p95, freshness y errores.
