Catalog (Overview)

TL;DR​

El módulo Catalog define el “qué” en VyteMerge: las definiciones reusables que un Profile ofrece.

  • Service = definición reusable de un servicio (consulta, masaje, corte, alquiler de cancha)
  • Product = definición reusable de un producto (shampoo, kit uñas, merch)

El Catalog no vende ni agenda por sí solo. Para vender o reservar se usa un Listing, que define el “cómo se ofrece” (precio, reglas, visibilidad, scheduling, etc.).

Contexto / Objetivo​

En la vida real, el mismo “qué” puede venderse en múltiples variantes:

Ejemplos​

Service: “Masaje”

  • estándar 60 min
  • “Masaje express” 30 min
  • promo “2x1”
  • pack “3 sesiones”
  • “a domicilio” (futuro)

Product: “Shampoo”

  • venta individual
  • pack x3
  • promo por canal
  • add-on al reservar un service (futuro)

Si mezclamos definición + venta + agenda en un mismo objeto, terminamos con:

  • duplicación de información (mismo servicio repetido 10 veces)
  • dificultad para actualizar (cambiar descripción/fotos en varios lugares)
  • inconsistencias (duración/precio/términos mezclados)

Catalog existe para separar lo estable (definición) de lo variable (venta/oferta).

Qué incluye el módulo Catalog​

1) Services​

Un Service representa la definición reusable de un servicio.

Incluye típicamente:

  • nombre y descripción
  • categoría y tags (descubrimiento)
  • duración base (lo normal)
  • media base (fotos/videos) como “default”
  • (opcional) términos base del servicio (genéricos)

Importante

  • La duración base es una referencia.
    Un Listing puede ajustar duración (ej: “express”) sin modificar el Service.

2) Products​

Un Product representa un bien vendible (físico o digital).

Incluye típicamente:

  • nombre y descripción
  • categoría y tags
  • media base
  • (futuro) variantes (talle/color/pack), stock, logística

3) Catálogo del Profile​

Cada Profile tiene un “catálogo” compuesto por:

  • Services
  • Products

Esto habilita:

  • organización interna del vendedor
  • reutilización en varios listings
  • analytics por item del catálogo (qué se vende más, qué convierte mejor)

Qué NO hace Catalog (límites claros)​

Catalog NO define:

  • precios ni promociones
  • visibilidad y distribución (public/followers/channels/partners)
  • scheduling, slots, buffers, capacity
  • bookings (reservas)
  • orders (compras)

Todo eso vive en:

  • Listings (cómo se ofrece/vende)
  • Bookings (reservas de servicios)
  • Orders (compras de productos)

Relación con Listings (regla mental)​

Un Listing siempre referencia un “origen” del catálogo:

  • Service Listing → referencia ServiceId
  • Product Listing → referencia ProductId

El Listing puede:

  • reutilizar la media base del Service/Product (fallback)
  • sobreescribir título/descripcion/media para una oferta puntual
  • definir precio, promos, reglas y canales

Diferencia útil:
Catalog = definición reusable
Listing = oferta comercial concreta

Integración con otros módulos (visión de negocio)​

Listings​

  • toma un Service/Product como base
  • define “cómo se vende” (incluye promos y rules)
  • es lo que aparece en feed/marketplace/campaigns

Timelines + Conflict Rules + Places​

En servicios reservables:

  • Timeline define el tiempo disponible
  • Conflict Rules determinan qué bloquea disponibilidad
  • Places definen “dónde” y zona horaria (cuando aplique)

Bookings / Orders​

  • Booking = reserva de tiempo (Service Listing)
  • Order = compra de items (Product Listing) y, en el futuro, combos

Tags​

  • Tags en Service/Product sirven para discovery global
  • Tags en Listing sirven para “cómo lo vendo” (promo, canal, etc.)

Reglas prácticas (MVP)​

  • Un Service/Product puede existir sin listings (todavía no se vende)
  • Un Listing siempre referencia un Service/Product (no hay listing “huérfano”)
  • Media:
    • si el listing no tiene media propia → usa media del service/product como fallback
  • Cambiar un Service/Product no debería romper bookings pasados:
    • los bookings guardan snapshot de lo relevante (desde Listings/Terms)

KPIs (para producto)​

(Opcional, pero útil para medir adopción)

  • Services creados por profile
  • Products creados por profile
  • % listings que reutilizan un mismo service (reuso)
  • conversiones por service/product (derivadas de listings/bookings/orders)

Checklist / Próximos pasos​

  • Documentar Service (propiedades mínimas y duración base)
  • Documentar Product (propiedades mínimas; dejar variantes/stock como futuro)
  • Definir políticas de media (fallback vs copy)
  • Alinear Tags: global vs profile-owned vs private
