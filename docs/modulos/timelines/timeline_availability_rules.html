Timeline Availability Rules

Coordinación de eventos, slots y acciones a nivel Timeline​

1. Visión general​

Timeline Availability Rules permiten definir cómo los eventos existentes impactan la disponibilidad futura y qué acciones ejecutar cuando ocurren conflictos.

Estas reglas:

  • Viven en el Timeline
  • Son independientes de un Offering específico
  • Se aplican a múltiples Offerings y Events
  • Son la base para automatizaciones y acciones futuras

A diferencia de Shared Capacity Rules (que viven en Offerings), estas reglas modelan conflictos de agenda y eventos externos.

2. Problema que resuelven​

En la práctica:

  • Un timeline puede representar una persona, familia o equipo
  • Existen eventos que bloquean tiempo (reuniones, viajes, actos escolares)
  • Algunos eventos afectan solo ciertos servicios
  • Otros afectan todo

Sin reglas claras:

  • El usuario no entiende por qué desaparecen slots
  • Se generan sobre-reservas o cancelaciones manuales
  • La lógica queda distribuida en múltiples lugares

Timeline Availability Rules centralizan esa lógica.

3. Principios clave​

3.1 Las reglas observan eventos, no personas​

  • No hay asignación directa de recursos
  • Solo se observan eventos ya existentes

3.2 El Timeline es la fuente de verdad​

  • Los Offerings no deciden
  • El Timeline evalúa y ejecuta

3.3 Separación clara de intención​

  • Slots → disponibilidad futura
  • Events → bookings existentes

4. Qué hacen estas reglas​

  • Observan eventos en uno o más timelines
  • Filtran por tags y status
  • Detectan solapamientos temporales
  • Ejecutan acciones sobre:
    • Slots
    • Events

5. Qué NO hacen​

  • ❌ No crean slots
  • ❌ No asignan staff
  • ❌ No modifican capacity
  • ❌ No crean Offerings
  • ❌ No deciden duración

6. Estructura general de una regla​

Toda Timeline Availability Rule se compone de tres secciones:

  1. Source – qué eventos se observan
  2. Target – qué se ve afectado
  3. Actions – qué hacer ante conflicto

7. Source (eventos observados)​

7.1 Timelines​

  • Timeline actual
  • Timelines propios
  • Timelines compartidos

Se pueden seleccionar múltiples timelines

7.2 Filtros de eventos​

Status​

  • Pending
  • Confirmed

Cancelled y Completed se ignoran por defecto

Tags​

  • #family
  • #travel
  • #school
  • #meeting

Solo eventos que cumplan todos los filtros participan.

8. Target (qué se ve afectado)​

8.1 Offerings​

  • Uno o varios Offerings específicos
  • O All Offerings

8.2 Events (opcional)​

  • Afectar solo eventos con ciertos tags
  • O afectar todos

Esto permite reglas como “solo cancela consultas presenciales”

9. Acciones​

Las acciones se dividen por tipo de entidad.

9.1 Acciones sobre Slots​

Hide slots (default)​

  • El slot no se muestra
  • No afecta bookings existentes

Notify me, I will decide​

  • El slot se muestra
  • Se genera una alerta

9.2 Acciones sobre Events​

Notify me, I will decide (default)​

  • No se cancela nada automáticamente

Cancel event​

  • Cancela el evento

Cancel and reschedule​

  • Cancela
  • Intenta encontrar nuevo slot

10. Detección de conflicto​

Una regla se activa cuando:

  • Existe un evento en el source
  • Cumple filtros
  • Se solapa en el tiempo
  • Considerando:
    • Duración del servicio
    • Buffers pre/post

11. Ejemplos reales​

Ejemplo 1 — Bloqueo total por agenda personal​

Source

  • Timeline: Personal
  • Tag: #travel

Target

  • All Offerings

Action

  • Hide slots

Resultado:
Durante viajes no se muestra disponibilidad.

Ejemplo 2 — Evento escolar afecta solo ciertos servicios​

Source

  • Timeline: Familia
  • Tag: #school

Target

  • Offerings: Consulta presencial

Action

  • Hide slots

Resultado:
Los servicios online siguen disponibles.

Ejemplo 3 — Monitoreo sin bloqueo​

Source

  • Timeline: Equipo
  • Status: Confirmed

Target

  • All Offerings

Action

  • Notify only

Resultado:
El usuario decide manualmente.

12. Convivencia con Shared Capacity Rules​

  • Timeline rules se evalúan antes
  • Si bloquean → capacity no importa
  • Si no bloquean → capacity decide

Orden mental:

  1. Timeline Availability Rules
  2. Shared Capacity Rules
  3. Mostrar slot

13. UX sugerido​

  • Tab “Availability” dentro del Timeline
  • Reglas listadas como frases legibles:
    • “If Family timeline has event tagged #school → hide slots for Presencial”
  • Preview visual del impacto

14. Por qué este modelo funciona​

  • Fácil de explicar
  • Escalable
  • No acopla Offerings entre sí
  • Permite automatización futura (SMS, email, webhooks)

15. Frase final​

Timeline Availability Rules coordinan eventos y decisiones.
No venden slots, aseguran coherencia.
